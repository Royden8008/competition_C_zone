# C 区(茂密森林)避障 —— 技术复盘

> 面向面试复盘。重点讲**问题诊断 → 根因定位 → 方案设计 → 工程取舍**这条技术主线;
> 调试方法只在末尾简述。

---

## 1. 背景与任务

- **平台**:ROS Noetic + PX4/MAVROS,Livox Mid360 激光雷达 + FAST-LIO(LIO 里程计与点云配准),机载 Jetson。
- **任务**:无人机自主穿过**乱序排列**的树林(树高约 2m、直径约 0.5m,相邻树缝约 1m),
  **飞行高度必须 <2m**(超高则该项 0 分),稳定飞过判分线得分。树位置赛前随机变更 → **不能背航点**。
- **几何约束**:树高 2m、限高 2m → **不能从树顶飞过,只能在树干间横向钻**,本质是一个
  **依赖实时感知的 2D 在线避障**问题。
- **机体尺寸**:机心→桨尖 0.30m(整机直径 0.60m),这是后续膨胀半径等参数的硬下限。

原有代码是校赛遗留的两层避障:

```
GridPlanner(2D 栅格 A*)      ── 点云→占据栅格→障碍膨胀→A* 规划绕障路径(感受野内)
        ↓ 输出中间航点(后退视界,执行若干点即重规划)
WaypointFollower(反应式避障) ── 沿路径飞,前向锥 + 周身安全球做减速/急停兜底
```

架构本身合理(全局规划 + 反应兜底,适合随机树林),但实测暴露出方法层缺陷。

## 2. 现象 → 诊断(本复盘的核心)

### 2.1 决定性现象

校赛实测时观察到两个现象:

1. **两棵障碍物沿无人机前进方向(x 轴)排成一条直线时,避不开;错开摆放时能避开。**
2. 长时间运行后,**点云地图出现漂移**(树重影/位置偏移)。

### 2.2 现象一的根因:单帧快照 + 无障碍记忆 + 激光遮挡

阅读源码后定位到结构性问题——**整套代码没有任何"障碍记忆"**:

- `GridPlanner::cloudCb` 与 `WaypointFollower::cloudCb` 都是 `cloud_ = c;`,**每来一帧点云直接替换**,旧帧丢弃。
- 规划与距离查询都用 `snap = cloud_`,即**只用最新一帧** `/cloud_registered`。
- 全工程检索 `local_map / octomap / esdf / accumulate`:无。`last_path_` 只是"路径不左右横跳"的
  迟滞,**不是障碍记忆**。

结合激光雷达的物理特性,这恰好解释"一条线避不了":

```
无人机 ──→  🌲树1  ····(树1的激光阴影)····  🌲树2
```

- 树2 在树1 正后方,**激光被树1 挡住,树2 在该帧里没有任何回波**;
- 代码只用这一帧、又无记忆 → **规划器根本不知道树2 存在**,只绕树1;
- 绕过树1 后树2 突然近距离出现 → 反应层急停 → `blocked → skip → replan`,但此时已进入急停距离
  → 卡死/超时,甚至来不及刹车撞上。
- **错开摆放**:两棵树互不遮挡,**同一帧里都有点** → A* 一次就织出绕行路径 → 能避。

> 这个"直线失败 / 错开成功"的对比,本质是一次**无意做出的遮挡判定实验**,结论很干净:
> 问题在"**看不到 + 不记忆**",不在避障算法本身。

补充并发症:**Mid360 是非重复扫描雷达,单帧本就稀疏**,需累积零点几秒才够密;单帧下细树干点数
偏少,又被聚类滤波(`cluster_min_points`)当噪点剔除,进一步加剧漏检。

### 2.3 现象二的根因:林中 LIO 漂移

树林**几何自相似、特征贫乏**,是 LiDAR-惯性里程计最易漂的场景:

- `/cloud_registered` 用**正在漂的位姿**配准 → 位姿一漂,点就摆错位 → 重影;
- 喂给飞控的位姿也漂 → 无人机"自以为在 A,实则在 B" → 朝树撞。

这与"无记忆"是**两个独立的洞**,但会叠加:地基(定位)在晃,上面盖什么都歪。

## 3. 方案设计:滚动局部障碍地图(Rolling Local Obstacle Map)

### 3.1 关键洞察:一对矛盾,一个折中

- 修遮挡 → 需要**记住**刚才看到的障碍(memory);
- 抗漂移 → 需要**忘掉**旧数据,以免漂移拖影累积(forgetting)。

两者方向相反,**用一个"有界滚动窗口"同时满足**:只保留最近一小段时间/空间的点。
这正是成熟无人机避障(局部 occupancy / ESDF 地图)的标准思路。

### 3.2 具体做法

在 FAST-LIO 与现有避障之间插入一个**独立节点** `local_obstacle_map_node`,只做点云层面的处理:

```
/cloud_registered ─→ [local_obstacle_map_node] ─→ /cloud_local_map ─→ GridPlanner / WaypointFollower
   单帧·稀疏·无记忆        累积 + 裁剪 + 去重         更密·有记忆·有界        (消费者代码零改动)
```

节点逻辑(每帧):

1. 单帧体素下采样;
2. 压入帧队列,**按帧龄(`max_age`)与帧数(`max_frames`)淘汰旧帧**;
3. 合并所有保留帧,**以无人机当前位置为心,裁掉 `max_range` 外的点**(用与点云同坐标系的
   `/Odometry` 取位置);
4. 合并结果再体素一次去重,以原 `frame_id` republish。

| 措施 | 解决的问题 |
|---|---|
| 累积最近 `max_age`(默认 1.5s)的帧 | ① 稀疏单帧变密,细树干不再被滤掉;② **记住被遮挡的树** → 修复"一条线避不了" |
| 按距离裁剪 + 限帧龄/帧数 | 把 LIO 漂移的拖影框在小范围,不让其累积成重影;同时控内存/算力 |
| 合并后再体素 | 去除重叠帧的重复点 |

### 3.3 工程取舍(面试可展开的点)

- **为什么做成独立 republisher 节点,而不是改进 GridPlanner/WaypointFollower 内部?**
  - 消费者(规划、反应层、降落区评估、守护下降)代码**零改动**,风险可控、便于回滚;
  - 累积逻辑只写一处,**所有消费者一致受益**;
  - 天然支持 **A/B 对照**:把 `cloud_topic` 切回 `/cloud_registered` 即退回原架构,
    可直接量化"局部地图带来的改善"。
- **为什么在世界系累积而非机体系?** `/cloud_registered` 已是世界/odom 系,直接拼接即可,
  且下游规划本就在该系工作,无需坐标变换;漂移问题交由"有界窗口"控制而非更复杂的机制。
- **参数即设计旋钮**:`max_age` 是核心——调大记得更牢(更抗遮挡)但漂移拖影更重,调小反之;
  本质是"记忆 vs 漂移"这对矛盾的显式调节钮。

### 3.4 落地形态

- 新增独立节点,launch 中用 `local_map`(默认开)开关;关闭时一条命令退回原始单帧。
- 全部参数(记忆时长、保留半径、体素、里程计话题)暴露为 launch arg,默认值即可起步。
- 不改任何规划/控制逻辑;现有工作空间编译通过。

## 4. 与避障参数的关系(顺序很重要)

地基补好后,**才轮到调上层参数**(否则是在晃的地基上贴瓷砖):

- 膨胀半径 `inflation_radius` ≥ 机半径(0.30)+ 裕度 → 取 **0.40**,保证 10cm 离桨间隙;
  树缝 1m 下机心走廊仍剩 0.20m。
- 反应层阈值排成梯队:`emergency < stop < relaxed膨胀 < 正常膨胀`,
  避免**规划层与反应层互相打架**(原配置 `stop_dist == relaxed_inflation` 即无裕度、易反复急停)。
- 限高靠 `cruise_height`(<2m,取树干中段如 1.0~1.2m)保证,**不是**靠障碍点过滤上限参数。

## 5. 调试方法(简述,非重点)

- **CSV 飞行日志**:节点内置 `flight_logger`,自主飞行时自动逐帧落盘(位姿、最近障碍距离、
  限速系数、关键事件),用于"撞没撞、卡在哪"(**量**)的定位,炸机也不丢。
- **rosbag 录包 + 离线复盘**:手动按需录原始雷达/点云/位姿,RViz 回放排查"定位漂不漂、树进没进
  点云"(**因**)。可改参数重放同一包。
- **A/B 对照**:`local_map` 开关一键切换新旧架构,量化改善。
- 排查顺序:**自底向上**(定位 → 点云 → 规划 → 反应 → 控制),底层坏了上层一定乱。

## 一句话总结

> 通过"两棵树一条线避不了、错开能避"的现象,定位到原架构**只用单帧点云、无障碍记忆**的方法硬伤
> (叠加激光遮挡与 LIO 漂移);用一个**有界滚动局部障碍地图**同时化解"记忆 vs 漂移"的矛盾,
> 以**独立 republisher 节点、消费者零改动、可 A/B 回滚**的方式低风险落地。
